🚗 从手机 App 到车载系统：一个 Android 开发者的“转向尝试”

羽晨sir 💕
  2026-03-18 13:48:47 2026-03-18 13:48:47 Created   2026-03-19 19:31:22 2026-03-19 19:31:22 Updated  
  • 学习 
 

最近在准备一些新的方向，发现一个挺有意思的赛道：车载 Android（IVI）。
于是我做了一个小 Demo，也踩了一些坑，这篇就当是一次“转向记录”。

一、为什么突然看车载？

说实话，一开始是面试逼的（笑）。

但稍微了解之后发现：

👉 车载并不是“把手机 App 放大”，而是完全不同的一套思路。

简单总结一句话：

手机是“让你用得爽”，车机是“让你别出事”。

二、我做了个什么 Demo？

我给自己定的目标很简单：

👉 不搞复杂系统，先模拟“车机中控的核心体验”

最终做了一个：

🚘 车载信息娱乐系统（IVI）Demo

包含几个模块：

  • 首页（类似车机桌面）
  • 音乐播放（支持后台播放）
  • 车辆信息（模拟数据）
  • 夜间模式（车载刚需）

三、先说结论：车载开发真的不一样

做完这个 Demo 后，我最大的感受是：

❗“少就是多”

在手机上你可以：

  • 多动画
  • 多层级
  • 多交互

但在车机上：

👉 你做的越多，用户越危险

四、模块拆解（稍微技术一点）

🎵 1. 音乐模块：我为什么用了 Service？

一开始我很天真，直接在 Activity 里写播放逻辑。

结果一切后台：

👉 音乐直接停了

然后我才老老实实用：

  • Service + MediaPlayer

结构大概是这样：

👉 关键点：

  • Service 保证后台播放
  • UI 不直接控制播放器（解耦）
  • 用 EventBus 做状态通知

🚘 2. 车辆信息模块：我为什么不用网络？

这里我一开始其实想接接口（显得高级一点）

后来查了一下资料才发现：

👉 真实车载数据很多来自 CAN 总线，而不是 HTTP 接口

所以我改成：

  • 模拟数据（速度 / 油量 / 里程）
  • 定时刷新（类似实时数据）

👉 虽然是假的，但思路是对的

🧱 3. 首页设计：被 UI 教做人

我一开始做的首页是这样的：

  • 小按钮
  • 多入口
  • 花里胡哨

然后我自己试了一下：

👉 根本点不准 😂

最后改成：

  • 大卡片
  • 少入口
  • 一屏搞定

👉 这时候才理解一句话：

车载 UI 本质是“降低操作成本”

🌙 4. 夜间模式：一个被低估的功能

手机夜间模式是“舒服”，
但车载夜间模式是“必要”。

实现很简单：

  • 两套主题（Day / Night）
  • 动态切换

但意义很大：

👉 夜间不刺眼 = 更安全

五、我踩过的几个坑（真实经历）

❌ 坑1：UI 刷新太频繁

一开始我每 100ms 刷一次车辆数据：

👉 直接卡顿

后来改成：

  • 降低刷新频率
  • 合并更新

❌ 坑2：模块耦合太高

一开始：

  • Activity 直接调播放
  • 播放直接改 UI

👉 后期改代码非常痛苦

后来用：

  • ViewModel + EventBus

👉 才稍微舒服一点

❌ 坑3：过度设计

一开始我甚至想：

  • 做导航模块
  • 接地图 SDK

后来我及时清醒：

👉 Demo 的核心是“表达能力”，不是“做产品”

六、车载 vs 手机（我的理解）

维度 手机 App 车载系统
目标 体验 安全
UI 精细 简洁
操作 多 少
稳定性 重要 极其重要

七、我目前的认知（不装）

我现在的水平：

  • ❌ 没做过真正车载项目
  • ❌ 没接触过底层（HAL / CAN）

但我已经开始理解：

👉 车载开发不是“技术难”，而是“场景不同”

八、下一步打算

接下来准备继续深入：

  • Android Automotive
  • 性能优化（ANR / 内存）
  • 更贴近真实车机环境

九、最后说点真实的

这次做 Demo 最大的收获不是代码，而是：

👉 意识到“不同场景的开发，本质是思维方式的变化”

如果你也在做 Android，建议你试试换个方向看看。

说不定会有新的思路。
