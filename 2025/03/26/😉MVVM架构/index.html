MVVM 架构在 Android 中的使用与对比分析（MVC / MVP）

一、什么是 MVVM？

MVVM（Model-View-ViewModel）是一种基于数据驱动的架构模式，核心思想是将 UI（View）与业务逻辑彻底解耦，通过 ViewModel 作为中间层，实现数据的自动同步与状态管理。

1. 结构组成

  • Model（模型层）
    • 负责数据的获取与存储（本地数据库、网络请求等）
    • 不关心 UI 展示
  • View（视图层）
    • 对应 Android 中的 Activity / Fragment / XML
    • 只负责展示数据，不处理业务逻辑
  • ViewModel（视图模型层）
    • 连接 View 与 Model
    • 持有 UI 状态数据
    • 处理业务逻辑，但不直接引用 View

2. 核心思想

  • 数据驱动 UI（而不是 UI 控制数据）
  • 使用观察者模式（如 LiveData / Flow）实现自动更新
  • 解耦 UI 和逻辑，提高可测试性与可维护性

二、MVVM 在 Android 中的实际使用

在 Android 开发中，MVVM 通常结合 Jetpack 组件实现：

  • ViewModel（官方架构组件）
  • LiveData / StateFlow（数据响应）
  • DataBinding / ViewBinding（UI绑定）

三、MVC、MVP、MVVM 对比

1. MVC（Model-View-Controller）

特点：

  • Controller 负责逻辑处理
  • View 和 Model 存在一定耦合

在 Android 中的问题：

  • Activity 既是 View 又是 Controller
  • 容易变成“上帝类”（God Object）

缺点：

  • 耦合严重
  • 难维护、难测试

2. MVP（Model-View-Presenter）

结构：

  • View（接口）
  • Presenter（逻辑处理）
  • Model（数据）

特点：

  • View 和 Presenter 通过接口通信
  • Presenter 持有 View 引用

示例：

优点：

  • 解耦较 MVC 好
  • 逻辑清晰

缺点：

  • Presenter 容易臃肿
  • 需要大量接口代码
  • 生命周期管理复杂（容易内存泄漏）

3. MVVM（Model-View-ViewModel）

特点：

  • ViewModel 不持有 View
  • 使用数据绑定（DataBinding / LiveData）
  • 数据驱动 UI

优点：

  • 解耦最彻底
  • 支持响应式编程
  • 易测试（ViewModel 可单测）
  • 生命周期安全（官方组件支持）

缺点：

  • 初学成本稍高
  • 需要理解响应式思想

四、三种架构对比总结

特性 MVC MVP MVVM
解耦程度 低 中 高
复杂度 低 中 中高
可测试性 差 好 很好
生命周期安全 差 一般 很好
代码量 少 多 中
推荐程度 ❌ 不推荐 ⚠️ 过渡方案 ✅ 推荐

五、MVVM 的优势总结

  1. 彻底解耦 UI 和业务逻辑
  2. 减少样板代码（配合 DataBinding）
  3. 更符合现代响应式开发（Flow / LiveData）
  4. 与 Jetpack 完美契合
  5. 天然支持单元测试
